Preface

Atlassian Confluence is a team workspace for collaborating on documents, knowledge bases, and wikis. It's ideal for sharing information, brainstorming, and creating a single source of truth. For confluence use cases, you can refer to the Atlassian documentation.

Before diving into the configuration steps, let's first explore the Atlassian Jira products supported by HYCU R-Cloud. To learn more about each supported SaaS application in the Atlassian ecosystem, please visit the SaaS application guides.

This article outlines the steps to configure and add Atlassian Confluence as an R-Cloud module with minimal required permissions. To protect your Atlassian Confluence data, you must add the R-Cloud Confluence module to HYCU R-Cloud as a source.

1.3 Permissions

To successfully add Atlassian Confluence as an R-Cloud module within HYCU, the Atlassian user must possess:

  • Confluence Admin Global permission
  • View All access at the Confluence space level

2. Permission Assignment

Company: A financial services firm heavily reliant on Atlassian Confluence for collaboration and documentation related to projects and issue tracking. All these documents are maintained within a Confluence space named "Repository."

Challenge: The company has implemented a strict security policy, enforcing the principle of least privilege. This policy prohibits default administrative user groups (Site or Global admins) from possessing Confluence admin permissions. Consequently, the user performing backups cannot maintain permanent restore (write) privileges, as this would violate the least privilege principle. Restore privileges should only be granted when a restore operation is necessary.

Solution: The company has decided to utilize dedicated Groups:

  • confluence-backup-admin: For performing Confluence data backups.
  • confluence-restore-admin: For performing Confluence data restores.

A dedicated user account named Liam Gray maintains the role of Confluence Admin. Importantly, neither of these groups will have any other Global or Project-level permissions.

Now, let's implement the solution. First, let's assign the following to the group named "confluence-backup-admin":

  • Space Access (View All) permission

Procedure

To configure the Confluence Product Admin role for the group, the logged in Org or Site admin must navigate to the User Management section.

  • Next, on the Directory page: Click on the "Groups" tab. Select the "confluence-backup-admin" group.

  • After following the steps outlined, the "confluence-backup-admin" user group will have the Confluence Product Admin role assigned to them within the Atlassian ecosystem.

  • Add the Liam Gray Service Account (user) as a member of the "confluence-backup-admin" group. The Service Account will inherit the Confluence Product Admin role.

  • Next, on the Global Permissions page, you will observe that the "confluence-backup-admin" group is associated with the "Confluence Admin" tag.

2.1.3 Assign Space Access Permissions

When a Confluence user lacks View Permission within a Space, they are effectively locked out of that specific Space. This means they cannot:

  • See the Space in their list of Spaces: The Space will be hidden from their view.
  • Access any content within the Space: This includes pages, blogs, attachments, and any other information stored there.

Therefore, it is crucial that the Atlassian group responsible for performing backups possesses View Permission within each Space that has to be backed up.

Procedure

  • Log in to Confluence. Find and click on the "Spaces" link, usually located in the navigation menu. Find and click on the "Repository" space from the list of available spaces.
  • In the "Repository" space page, navigate to: Space Setting ‚Üí Settings ‚Üí Space access ‚Üí Groups.

  • Next, on the Space access page, click Edit and Enable View (All) Permission for the confluence-backup-admin group.

Note üëâ

However, this process can be easily scaled to encompass multiple Spaces by utilizing the "Add people to all spaces" button, which allows for bulk permission assignment. For further information, please refer to the relevant Atlassian article.

2.2 Restore Permissions

In the previous steps, you only provided minimal permission (View) on a Confluence Space for HYCU to perform a backup. However, View permission alone is insufficient for restoring Confluence data. The user or group performing the restore must have "Add permission" within the Space.

2.2.1 Approaches

Here are two approaches to address this requirement:

1. Granting Additional Permissions to the Backup Group:

  • Grant Add permission to the existing "confluence-backup-admin" group within the Space. This approach simplifies management, but may increase security risks as HYCU will have restore (write) permissions at all times.

2. Granting Additional Permissions to the dedicated Restore Group:

  • Create a new dedicated group (e.g., "confluence-restore-admin").
  • Grant Add permission to this new group within the Space.
  • Temporarily add the backup user(s) to this group only when a restore is required.
  • Remove the user(s) from the restore group once the restore is complete.

This second approach provides a more secure and granular control over restore permissions by limiting access to a specific group and only granting it when necessary. We will describe the latter procedure in more details.

Problem:

  • HYCU relies on Atlassian user accounts for backups.
  • Due to Atlassian throttling, a single Atlassian user account can become a performance bottleneck in large-scale environments.

Solution:

Adding multiple service accounts in HYCU can distribute the backup load, improving performance and efficiency. The recommendation is to add 3 service accounts as a starting point.
